## 功能任务优先级区域

#### 🔴 Task 1: 运动控制与 TF 追踪任务 (Priority 4 - 最高优先级)

* **执行频率：** 严格 100Hz (10ms 周期，使用 `vTaskDelayUntil`)
* **核心职责 (反射神经与肌肉)：**
    1.  **极速防撞：** 检查雷达正前方探测数据，若小于危险阈值，直接触发硬件刹车（覆盖所有高级指令）。
    2.  **TF 坐标变换：** 将 `GlobalPath` 中的下一个目标点，结合 `global_robot_state` 当前位姿，实时转换到底盘局部坐标系。
    3.  **轨迹跟踪：** 运行纯追踪算法 (Pure Pursuit)，算出期望速度 $v$ 和 $\omega$。
    4.  **底层闭环：** 读取编码器，运行 PID 速度环，输出 PWM。
* **算力消耗：** 极低 (\< 1ms)。必须快进快出，绝不恋战。

#### 🟠 Task 2: IMU 姿态解算任务 (Priority 3 - 极高优先级)

* **执行频率：** 严格 200Hz (5ms 周期，使用 `vTaskDelayUntil`)
* **核心职责 (前庭系统)：**
    1.  读取 IMU 原始数据 (SPI/I2C)。
    2.  硬件 FPU 加速解算四元数与偏航角 (Yaw)。
    3.  **进入临界区**，极速更新 `global_robot_state.yaw` 和 `yaw_rate`，退出临界区。
* **算力消耗：** 极低 (\< 1ms)。

#### 🟡 Task 3: 雷达 DMA 路由任务 (Priority 2 - 高优先级)

* **执行频率：** 10Hz (由串口 IDLE 中断触发唤醒，非阻塞)
* **核心职责 (视觉视神经过滤)：**
    1.  校验雷达乒乓 Buffer 中的一帧数据。
    2.  读取当前角速度，执行\*\*“运动学跳帧决策”\*\*（直行 10 帧发 1 帧，转弯逐帧发送）。
    3.  将需要处理的 Buffer 指针投入 `Lidar_Queue`，并从空闲队列取回新指针交还给 DMA。

#### 🟢 Task 4: 算法综合大脑任务 (Priority 1 - 最低优先级)

* **执行频率：** 尽力而为 (Event-Driven，平时休眠，收到雷达指针被唤醒)
* \**核心职责 (大脑皮层：ICP, SLAM, A*, 蓝牙)：\*\*
    1.  **ICP 匹配：** 结合 IMU 给出的 Initial Guess，计算高精位姿，修正 `global_robot_state`。
    2.  **双层地图更新：** 将新障碍物写入 `slam_map_2cm`，并 $O(1)$ 映射到 `nav_map_20cm`。
    3.  **蓝牙 DMA 触发：** 将地图更新坐标压入 TxBuffer，写满 50 个点后触发 DMA 蓝牙发送。
    4.  \**按需 A* 重规划：\*\* **仅当**发现新障碍物且该障碍物挡在了当前 `GlobalPath` 上时，暂停一切，运行 A\* 算法，拉线平滑，然后发送任务通知给 Task 1 换路。
* **算力消耗：** 极高 (20ms - 60ms)。它会在后台被 Task 1 和 Task 2 疯狂切片抢占，但由于机制安全，最终会在下一次雷达数据到来前算完。

# 🤖 NoeticMaze 机器人底层控制系统架构 (RTOS 版)

## 📌 架构概览

本项目基于 FreeRTOS 构建，采用严格的**速率单调调度 (Rate Monotonic Scheduling, RMS)** 原则。系统被划分为 5 个核心任务（Task），优先级严格按照任务的实时性要求和执行频率进行划分，以确保底层运动控制的绝对平滑与传感器数据的零丢失。

### 核心任务清单

| 任务名称 | 优先级 | 频率/触发方式 | 核心职责 |
| :--- | :---: | :--- | :--- |
| **Task 1: 驱动控制** | `High` (5) | 100Hz 严格周期 | 纯追踪轨迹跟随、PID 闭环、极速防撞 (10ms 级响应)。 |
| **Task 2: IMU 解算** | `Above_Normal` (4) | 200Hz 严格周期 | 硬件 FPU 解算姿态、更新全局位姿。 |
| **Task 3: 雷达 DMA** | `Normal` (3) | UART IDLE 事件 | 接收、校验雷达乒乓 Buffer，运动学跳帧分发。 |
| **Task 4: SLAM 核心** | `Below_Normal` (2) | 队列事件触发 | ICP 匹配更新位姿、写入全局地图、监督现有路径。 |
| **Task 5: A* 规划** | `Low` (1) | SLAM 唤醒触发 | 接收代价地图，后台计算全局最优路径。 |

---

## ⚙️ 核心线程协同机制与数据流

本系统在不使用高开销互斥锁（Mutex）的前提下，通过精妙的内存读写权限划分和指针传递机制，实现了复杂算法与高频控制的安全解耦。重点体现在 **Task 1 (小脑)**、**Task 4 (感知中枢)** 和 **Task 5 (大脑)** 的深度互动。

### 1. 局部极速防撞 —— 单写多读 (SWMR) 机制
* **数据载体：** `slam_map_2cm` (全局高精度栅格地图，2-bit 压缩)。
* **交互逻辑：**
  * **写权限绝对隔离：** 只有 **Task 4 (SLAM)** 拥有对该地图的修改权限。利用 Cortex-M4 架构特点，保证了读改写操作的内存级安全。
  * **无锁极速读取：** **Task 1 (驱动)** 在其 100Hz 的控制周期内，直接无锁读取车体前方的若干栅格进行防撞校验。即使 SLAM 正在写入，由于仅涉及局部环境，微小的“数据撕裂”对 10ms 级别的物理防撞毫无影响。这保证了底层控制极高的实时性。

### 2. 事件驱动重规划 —— 乒乓双缓冲机制
系统不会在每次环境改变时都盲目重规划，而是由 SLAM 充当“守门员”。
* **监督（SLAM 端）：** Task 4 在每次更新完 `slam_map_2cm` 后，会交叉比对当前的 `GlobalPath`。
* **触发条件：** **只有当**新发现的障碍物阻挡了当前正在行驶的路径时，Task 4 才会触发重规划动作。
* **指针移交：** Task 4 从 `Free_Queue` 获取一个空闲的缓冲区（`nav_map_10cm`），进行降采样和膨胀处理后，将该缓冲区的指针推入 `Astar_Queue`，唤醒 Task 5。
* **后台寻路（A* 端）：** Task 5 醒来后，在完全属于自己的私有缓冲区内执行耗时的 A* 算法，绝不会被底层的地图更新干扰，保证了规划数据的一致性。

### 3. 动态环境适配 —— 抢占式放弃机制 (Preemptive Abort)
针对 A* 算法在复杂死胡同中耗时过长的极端情况，系统设计了主动干预机制：
* 如果在 Task 5 (A*) 还在旧地图上苦苦寻路时，Task 4 (SLAM) 发现了新的路况变动，且 `Free_Queue` 已无空闲缓冲区。
* Task 4 将立起全局中断标志 `g_astar_abort_request = true`。
* Task 5 在其 `while` 搜索循环中嗅探到该标志，将立即丢弃当前计算，清理内存池，并归还缓冲区。
* Task 4 拿到归还的缓冲区后，压入最新鲜的地图并重新唤醒 Task 5。这确保了宝贵的 CPU 算力永远只用于解决“当下的问题”。

### 额外问题解答：能否把 SLAM 线程和 A* 线程设置为相同的优先级？

**强烈建议不要这么做。** 保持 A* (Task 5) 的优先级**低于** SLAM (Task 4) 是这套架构能够稳定运行的基石。

如果将它们设置为相同优先级（例如同为 Priority 2），在 FreeRTOS 中会引发以下致命问题：

**1. 时间片轮转 (Time Slicing) 导致 SLAM 丢帧**
当两个任务优先级相同时，FreeRTOS 会采用时间片轮转调度。这意味着 A* 这个计算密集型任务（动辄几十毫秒）会和 SLAM 平分 CPU 时间。
* **后果：** SLAM 原本可以在 10ms 内处理完一帧雷达数据，现在因为被 A* 不断抢占，处理时间可能被拉长到 20ms 甚至 30ms。这会直接导致下一帧雷达数据到来时，SLAM 还没算完上一帧，引发**严重的传感器丢帧和位姿漂移**。

**2. “抢占式放弃”机制失效 (Race Conditions)**
我们设计的架构精髓在于：SLAM 拥有最高的情报权，它可以随时“杀掉”过期的 A* 任务。
* 如果优先级相同，当 SLAM 发现新障碍物想要设置 `g_astar_abort_request = true` 时，它可能正好用完了时间片，被操作系统切走。接着 A* 带着旧地图继续盲目计算。这种平权状态会破坏“环境更新优先于路线规划”的逻辑链路，导致机器人在动态障碍物面前反应迟钝。

**3. 违背 CPU 算力压榨原则**
A* 算法的本质是一个“尽力而为 (Best-Effort)”的后台任务。我们希望 Cortex-M4F 在完成了所有关乎生死的实时任务（底盘闭环、防撞、解算位姿、收发雷达）之后，把**所有剩下的边角料时间**全部倾注给 A*。只有把 A* 放在最低优先级，才能实现这种完美的“零闲置、不逾矩”的算力压榨。
